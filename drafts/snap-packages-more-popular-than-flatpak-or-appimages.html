Are Snap way ahead of FlatPaks or AppImages?

Posted on Sun 13 January 2019 in Linux

On the Linux desktop there is a push for a universal packaging format. There appears to be three main containders: Snap packages, supported by Canonical and Ubuntu; FlatPak, promoted by the Gnome Foundation although not a Gnome project or Gnome specific; and AppImage, a community project. There are many technical differences between the three formats but they share some goals:

  • Distribution agnostic packaging: The packages are meant to run on any Linux distro be it Ubuntu, Fedora, OpenSUSE or even ArchLinux, ... . For developers this means package once run anywhere. To achieve this these package formats tend to bundle all the dependencies into the package although Snap and FlatPak allow the packages to rely on some shared runtimes.
  • Upstream packaging: Ideally all packages are made by the developers of the project as opposed to having maintainers (usually per distribution) ensuring the distribution of the software. The idea is that users use exactly what the developers made. In principle this entails the software users get is the latest version supported by the developers.

I will not comment on whether UPs are a good thing or not. However many places on the internet state Snap is way ahead of Flatpak or AppImage. Canonical states there are 4100 snap packages currently available. Whereas according to FlatHub there are 480 FlatPaks. Finally according to AppImageHug there are 634 AppImages. Many websites on the internet cite these statitics (albeit slightly now out of date). For instance see Verummeum.

Snap packages FlatPaks AppImages
< 4100 snaps 480 flatpaks 634 appimages

Is this true? Yes probably, however it is misleading.

Those numbers are misleading

Relying on those numbers to claim Snaps are taking over is misleading. It is misleading for two reasons. One we aren't comparing apple to oranges. And two it tells us little about what kind of software is being distributed as Snaps, FlatPaks or AppImages.

Snaps, FlatPaks and AppImages are the same thing

The first big different between these formats are not distributed in the same way. Snaps have only one store, whereas FlatHub isn't the only store for FlatPaks and AppImageHub is a recent effort to list all existing software distributed as AppImages. It goes without saying the number of FlatPaks and AppImages could be underestimated.

Bear in mind the differences do not stop there. AppImages are intented as a packaging format for end-user applications of the Desktop or the Server, but not for system level components. FlatPaks are desktop focused. Meanwhile Snaps have a more broader scope, being intented for Server and the Desktop and for both system and end-user software. Naturally having a broader scope means one should expect more software being packaged for Snap than for FlatPak or AppImage.

However neither of these reasons are the true reason why the comparaison shown in the table above is misleading. The bigger problem is what is being currently packaged as AppImage, Flatpak and Snap. Many applications made available on those platforms aren't relevant.

Problems in the software selection

By searching those hubs and stores I found many issues in what is being made available.

Snap store

Unfortunately the official website for the Snap store limits how one can explore the software selection. Thankfully the alternative uAppExplorer exists. Although it is hard to tell how well the two match (because uAppExplorer claims 2000 applications while Canonical claim 4100 applications). Anyway here are a couple of problems I found with the software selection on the Snap store:

  • A huge number of applications for non Intel/AMD architecture. For instance for ARM architure there 1176 packages out of 2007, in other words 58.6 percent. Also for instance we have 244 out of 2007 for IBM's System Z, about 12.2 percent; and 273 out of 2007 for PowerPC, about 13.6 percent. (In total 84.3%). Not exactly the typical architectures used by a typical Linux desktop user. In practice the typical Linux desktop user has about 781 snaps available on his/her platform, a number far closer to the number of available AppImages
  • A huge number of Playground applications, who serve no end-user purpose, the only point of those applications is that developers learn on to use and package Snaps. For instance if one searches on Snap store the keyword Hello one finds nearly 300 Hello applications. There are other examples, if one searches for Test one finds at least 100 Test applications. Of course one could image other similar key words give many results. However this means that by a conversative estimate about 10% of Snaps aren't real end user applications.
  • If one adds up the software made available on the Snap store in the different suggested sections (namely Dev, Games, Social, Crypto, Video, Music, Productivity, Utilities, Graphics and Server), I found there are about 400 currently applications, putting Snap behind both FlatPak and AppImage. Now of course many applications probably fit neither of offered categories but it highlights that the number of software made for end user must be far less than 4100.
  • Duplicates applications are present in the store. For instance Stellarium, Brackets, rsync and tmux are duplicated. Unfortunately I have no easy way of counting the number of duplicates present.

I simply don't think we should take the claim 4100 Snaps available seriously. Yes there might be 4100 but in practice the number of software one might install is far smaller.

FlatHub

I have fewer quicks with FlatHub. I did not find any Hello application and only 4 Test applications. When summing up the number of software available in each category I found at the time 527, which is slightly higher than the claimed at the time number of applications 479 (meaning some software is in multiple categories).

FlatHub does not provide information about the architecture. However to my understanding every application specify the supported architectures in a json file, this should mean an application available for multiple platforms is onyl counted once.

I did find one duplicate application namely Meteo.

Overall I trust the number provided by FlatHub.

AppImageHub

Again like FlatHub I found no Hello and I found no Test applications. Because all application are explictly listed there are no hidden application increasing the count. I have no information about architecture but there are no duplicates.

The number is indeed trustworthy, however it does not tell us exactly how many application are distributed as AppImage. Many projects could be distributing today AppImages and not be on the list.

Measuring success with existing popular applications?

Instead of focusing on how many packages there are out there, let's ask how many popular software projects have adopted either formats. Answering this question won't give us the full picture but it will give us an idea of whether we should expect using more of Snaps, FlatPaks or AppImages in the future.

Disclaimer: Now I must be honest, answering this question is difficult and easily subjective. I tried being impartial and accurate but you should take everything I state with a grain of salt. Try researching on your own.

Because counting on my own whether each piece of software on Linux is or not available as a Snap, as a FlatPak or as an AppImage is time consuming, I decided to take a subset of populat Linux applications. To avoid putting bias in the list I choose the set of software listed on FossMint. I excluded Google Drive and Evernote as they are not official packaged for Linux.

Using this list I searched for the following:

  • Are the projects officially using AppImage, FlatPak or Snap as a mean of distribution? In each case I checked the project's official website.
  • Are the software distributed either as AppImage, FlatPak or Snap? Regardless whether the project official supports either format.
  • When distributed as an AppImage, FlatPak or Snap is it the latest version?

I found for the software in the list the following statistics:

  • About 14% of projects official distribute their software either as as AppImage, FlatPak or Snap. Meaning that adoption is slow as each packaging format is already at least four years old. Of the 14% ~ 10 of applications which adopted one of the formats, 40% ~ 4 have adopted FlatPak, 40% ~ 4 have adopted AppImage and 20% ~ 2 have adopted Snap. However because the sample of official distributed AppImages, FlatPaks and Snaps is so small, one should not conclude yet that FlatPak or AppImage is preferred by software projects.
  • Concerning software being available as either formats (but not necessarly distributed by the project). Whilst AppImage appears to be last with about 37%, neither Snap nor FlatPak are clearly ahead with respectively about 46% and 47% adoption. This discrepancy can be explained by the AppImage project's insistence on upstream packaging.`.
  • Concerning whether we have the latest version packaged as either format we see a big lead for Snap. About 60% of Snaps provided ship the latest version, 53% of FlatPaks provided ship the latest version and only 37.5% of AppImages ship the latest version. AppImage has such a low number because most AppImages have been created by probonopd one of the main developers behind the AppImage project, and he has not been updating them for more than a year probably because of the insistence on upstream packaging. The numbers are healthier for Snap and FlatPak but it is still far from 100%. The reason for this is that most Snaps and FlatPak are shipped by volunteer maintainers, and some of them are no longer updating them.

Based on this list I don't think one can claim Snap is way ahead, if anything it appears to be strongly in competition with FlatPak. One should not conclude AppImage is losing steam either, AppImage is doing well with 634 applications distributed as AppImage according to AppImageHub and many developers choosing AppImage as the official format for Linux.

I decided to dig a bit further and see where each format is doing well:

  • Based on the list I used, I found FlatPak does well with GNOME applications, which should not be surprising because GNOME is promoting FlatPak for instance by making FlatPak one of the targets supported by GNOME Builder. Looking at FlatHub I was surprised to see a huge success of the format with games with about 132 games on FlatHub against 60 on Snapcraft.io, meanwhile AppImage has 40.
  • Based on AppImageHub and the list I used, AppImage does well with Electron and QT/KDE applications, in part because the default target for Linux with Electron Builder is AppImage and the packaging tooling for AppImage for QT is very mature.
  • Snap is more over the place. However unlike the other formats Snap is showing success with big vendors such as Microsoft, Mozilla or JetBrains. I think Canonical's marketing is doing well.

What holding back either Snap, FlatPak or AppImage?

Because the adoption rate is still small, a useful question is to understand why developers may choose to avoid any of those formats. I searched various forums and GitHub for clues as why developers are not choosing these formats. Here are some of the main reasons:

  • Lack of time, ressource or motiviation: This is one of the main reasons for not seeing a package format being adopted. See for instance the following links 1, 2, 3, 4, ... . This should not be surprising because most software are developed by hobbyists and they don't necessarly have the time to learn/developed for a new packaging format.
  • Not even packaged: I was surprised to see lots of software projects not even packaging on Linux to begin with (for instance rsync and tmux). For those projects distributions are supposed to have maintainers who package the software.
  • Software no longer developed: Lots of software are no longer being developed. For instance Frozen Bubble was last updated in 2010 whilst Banshee was last updated in 2014. It is hard to expect projects with no developers to magically creat a Snap, FlatPak or AppImage.
  • Lack of agreement over which format to use: This is not frequent but fairly telling. If Snap, FlatPak or AppImage are supposed to be the Universal Package Format how come we have 3 of them. RStudio could not agree over which format to use.

Lots of people in the Linux community criticizes Snap, FlatPak and AppImage on technical ground and disgree with the philosophy. But ironically this didn't show up much as a reason not to support Snap, FlatPak or AppImage. I think the people complaining about universal packaging are usually not the same as the software developers, I think criticism comes more from system administrators, Linux distribution developers/maintainers and users who choose Linux for its security.
